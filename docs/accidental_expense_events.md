# 意外破财事件机制

## 目标

“意外破财”用于替代传统“小偷偷钱”事件。它不强调随机被偷，而是模拟现代城市交易局里更贴近杭州语境的现金损耗：平台规则、临时费用、收款风险、电诈、保证金骗局、投资群回撤和违约赔付。

设计目标：

- 增加每次换地点后的不确定性和刺激感。
- 迫使玩家保留一定现金安全垫，避免永远满仓梭哈。
- 让破财事件和当前商品体系、平台经济、金融品类保持同一时代感。
- 控制损失上限，避免低概率事件直接毁掉整局。

## 触发时机

事件在每次换地点的一日结算流程中触发，位于：

1. 刷新市场价格
2. 结算欠债和存款利息
3. 收取仓库管理费
4. 处理市场新闻事件
5. 处理健康事件
6. 处理意外破财事件
7. 新手缓冲和临界状态检查
8. 消耗 1 天

当前实现入口为 `GameEngine.doStealEvents()`，虽然函数名仍沿用旧命名，但语义已经改为“意外破财”。

## 事件字段

每个事件配置包含：

| 字段 | 含义 |
| --- | --- |
| `freq` | 触发频率参数。每次换地点时执行 `rnd(1000) % freq === 0` 判定。 |
| `msg` | 事件文案，会写入事件日志。 |
| `ratio` | 按当前现金计算的损失比例。 |
| `min` | 本事件最低损失额。 |
| `max` | 本事件最高损失额。 |
| `severity` | 事件强度，当前为 `light` / `medium` / `heavy`。 |
| `debtOnShortfall` | 现金不足时，是否把差额转成欠债。 |
| `fame` | 可选，事件对名声的影响。 |

## 损失计算

触发某事件后：

```text
byRatio = floor(cash * ratio / 100)
desiredLoss = clamp(byRatio, min, max)
paid = min(cash, desiredLoss)
cash -= paid
shortfall = desiredLoss - paid
```

如果事件配置了 `debtOnShortfall`，并且现金不足：

```text
debtAdded = floor(shortfall * 1.15)
debt += debtAdded
```

如果事件配置了 `fameDelta`，对应当前实现里的 `fame` 字段：

```text
fame = clamp(currentFame + fameDelta, 0, 120)
```

现金最后会被保护到不低于 0。

## 新手缓冲

早期局面如果玩家状态较差，会降低破财伤害。

触发条件：

```text
nextDayNumber <= STARTER_BUFFER_DAYS
score < 0
cash < 2500
```

缓冲效果：

- `ratio` 降为原值约 55%，最低为 4%。
- `max` 降为原值约 45%，最低 500。
- `min` 降为原值约 45%，最低 80。
- 日志里追加“首周缓冲”。

## 反馈与记录

触发后会生成：

- 一条 `expense_event` 事件日志。
- 一条 `lastMarketPopups` 弹窗消息，标题为“意外破财”。

事件日志 payload 包含：

| 字段 | 含义 |
| --- | --- |
| `target` | 当前为 `cash`。 |
| `ratio` | 实际生效比例。 |
| `original_ratio` | 原始配置比例。 |
| `loss` | 实际扣除现金。 |
| `desired_loss` | 本应承担的损失。 |
| `debt_added` | 现金不足时新增欠债。 |
| `buffered` | 是否应用新手缓冲。 |
| `severity` | 事件强度。 |
| `source` | 原始事件文案。 |

## 平衡目标

当前模拟结果：80 局、每局最多 45 天，平均每局约 2.86 次意外破财事件，未出现现金负数。

建议调参区间：

- 平均每局 2 到 4 次较合适。
- 轻度事件应更常见，但不会改变整局方向。
- 中度事件应让玩家感到“现金安全垫很重要”。
- 重度事件应低频，但足以制造记忆点。
- 破财事件不应替代市场波动成为主要输赢来源。

## 可扩展方向

后续可以扩展为更细的事件类型：

- `target: "cash"`：扣现金。
- `target: "bank"`：影响存款或冻结存款。
- `target: "inventory"`：损坏、丢失、强制折价处理部分持仓。
- `target: "debt"`：产生赔偿、分期、垫付。
- `condition`：按地点、商品类型、现金规模、名声、健康、是否满仓触发。

也可以给事件增加“反制”空间：

- 名声高时，大客户违约赔付降低。
- 银行存款较多时，部分现金风险降低，但风控事件概率提高。
- 购买情报或做慈善后，某些电诈/保证金事件概率下降。
